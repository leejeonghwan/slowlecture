RAG 검색증강생성 뉴스 자동화 벡터 검색

RAG(검색증강생성)란 무엇인가요.

RAG(Retrieval-Augmented Generation, 검색증강생성)는 대언어모델(LLM)이 외부 지식을 검색해서 답변하는 기술입니다. 기존의 LLM은 학습 데이터의 시점에서 고정된 정보만 알고 있지만, RAG는 최신 자료를 실시간으로 검색하여 더 정확하고 신뢰할 수 있는 답변을 생성합니다.

질문 입력
→
지식 검색
→
LLM 처리
→
정확한 답변

기자·PD가 RAG를 필요로 하는 이유.

  • 정확한 출처 기반 답변. 국회 회의록, 판결문, 보도자료 등 공식 자료를 검색하여 팩트 기반의 기사를 작성할 수 있습니다.
  • 팩트체크 자동화. 수백만 건의 보도자료와 공식 기록에서 모순되는 정보를 자동으로 찾아낼 수 있습니다.
  • 아카이브 검색 효율화. 신문사의 기존 기사 데이터베이스에서 관련 기사를 즉시 검색하여 심화 보도를 준비합니다.
  • 다중 소스 통합. 선거자금 보고서, 정치자금 공시, 판결문 등 다양한 형식의 공공 데이터를 한 번에 검색합니다.
  • 실시간 정보 반영. 최신 보도자료, 성명서, 공시 정보를 즉시 기사에 반영합니다.
RAG의 기본 아이디어. LLM의 강력한 이해와 표현 능력 + 검색 기술의 정확한 정보 추출 = 신뢰할 수 있는 뉴스 자동화 시스템.

RAG의 전체 파이프라인.

RAG 시스템은 6개의 핵심 단계로 구성됩니다.

1. 문서 로드
↓
2. 청크 분할
↓
3. 임베딩
↓
4. 벡터 저장
↓
5. 의미 검색
↓
6. LLM 답변

1단계: 문서 로드.

기자·PD가 사용할 수 있는 문서 소스입니다.

  • 국회 회의록 (국회도서관 공개 자료)
  • 판결문 (대법원 판례 검색 시스템)
  • 보도자료 (각 부처, 기관, 기업)
  • 신문사 기존 기사 아카이브
  • 선거자금 보고서, 정치자금 공시
  • 통계청 보도자료 및 통계표
  • 정부 보도자료 (공식 보도자료 시스템)

2단계: 청크 분할.

긴 문서를 LLM이 처리 가능한 작은 단위로 나누는 과정입니다. 예를 들어, 100페이지 탐사보도 원고를 500-1000 토큰 크기의 청크로 분할합니다.

3단계: 임베딩.

각 청크를 벡터(수의 배열)로 변환합니다. 의미가 유사한 청크는 벡터 공간에서 가까이 위치합니다.

4단계: 벡터 저장.

변환된 벡터를 벡터 데이터베이스에 저장합니다. ChromaDB, Weaviate, Milvus 등을 사용할 수 있습니다.

5단계: 의미 검색.

사용자의 질문을 벡터로 변환한 후, 저장된 벡터 중 가장 유사한 청크들을 찾습니다.

6단계: LLM 답변.

검색된 관련 정보와 사용자의 질문을 함께 LLM에 전달하여 최종 답변을 생성합니다.

실제 예제: 국회 회의록 검색 시스템.

국회 회의록에서 특정 의원의 발언을 검색하고 요약하는 RAG 시스템을 구축해봅시다.

Python
# 필요한 라이브러리 임포트
from langchain.document_loaders import TextLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain.embeddings import OpenAIEmbeddings
from langchain.vectorstores import Chroma
from langchain.llms import OpenAI
from langchain.chains import RetrievalQA

# 1단계: 국회 회의록 문서 로드
loader = TextLoader("parliament_records.txt", encoding="utf-8")
documents = loader.load()

# 2단계: 청크 분할
text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=800,
    chunk_overlap=100
)
chunks = text_splitter.split_documents(documents)

# 3-4단계: 임베딩 및 벡터 저장
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
vectorstore = Chroma.from_documents(
    chunks,
    embeddings,
    persist_directory="./parliament_db"
)

# 5-6단계: RAG 체인 구성
llm = OpenAI(temperature=0, model_name="gpt-3.5-turbo")
qa_chain = RetrievalQA.from_chain_type(
    llm=llm,
    chain_type="stuff",
    retriever=vectorstore.as_retriever()
)

# 질의 실행
query = "의료보험 개혁에 대해 김○○ 의원이 한 발언을 요약해줘."
result = qa_chain.run(query)
print(result)
구현 팁. 국회도서관 국정감사 자료실이나 대법원 판례 검색 시스템의 공개 API를 활용하면, 자동으로 최신 자료를 수집하고 벡터 DB를 업데이트할 수 있습니다.

  • 사용 사례. 기사 작성 보조, 배경 정보 검색, 유사 기사 찾기.

기법 3: 신뢰할 수 있는 RAG (Reliable RAG).

RAG 시스템이 거짓 정보(할루시네이션)를 생성하지 않도록 보장하는 기법입니다. 팩트 기반 저널리즘에 필수적입니다.

Python
# 신뢰할 수 있는 RAG 구현
from langchain.chains import RetrievalQA
from langchain.prompts import PromptTemplate

# 답변 검증 프롬프트
verification_prompt = PromptTemplate(
    input_variables=["question", "context", "answer"],
    template="""주어진 문맥(context)에 기반하여 다음 답변이 정확한지 판단하세요.

문맥:
{context}

질문: {question}

답변: {answer}

판단:
1. 답변이 문맥에서 나온 정보만 사용했는가?
2. 답변에 문맥에 없는 정보(할루시네이션)가 포함되어 있는가?
3. 출처를 명확히 제시했는가?

결론: [정확함/부분 정확/거짓 정보 포함] 및 이유를 설명하세요."""
)

# RAG 체인에 검증 추가
class ReliableRAG:
    def __init__(self, vectorstore, llm):
        self.vectorstore = vectorstore
        self.llm = llm

    def query(self, question):
        # 1. 관련 문서 검색
        docs = self.vectorstore.similarity_search(question, k=3)
        context = "\n\n".join([doc.page_content for doc in docs])

        # 2. 답변 생성
        answer = self.llm.predict(context=context, question=question)

        # 3. 답변 검증
        verification = self.llm.predict(
            context=context,
            question=question,
            answer=answer
        )

        # 4. 출처 표시
        sources = [doc.metadata.get("source", "알 수 없음") for doc in docs]

        return {
            "answer": answer,
            "verification": verification,
            "sources": sources
        }
중요 주의사항. RAG는 할루시네이션(거짓 정보 생성)을 줄일 수 있지만 완전히 없앨 수는 없습니다. 반드시 생성된 답변을 원본 자료로 다시 확인하는 단계가 필요합니다.

기법 2: 명제 기반 청킹 (Proposition Chunking).

문서를 문법적 또는 의미적 경계로 자연스럽게 나누는 대신, 각 문장이나 단락을 하나의 "명제"(주장, 사실)로 분해하는 기법입니다.

Python
# 명제 기반 청킹 구현
from langchain.llms import OpenAI
from langchain.chains import LLMChain
from langchain.prompts import PromptTemplate

# 명제 추출 프롬프트
proposition_prompt = PromptTemplate(
    input_variables=["text"],
    template="""다음 텍스트를 간단하고 명확한 명제들로 분해하세요.

{text}

1.
2.
3.
..."""
)

        self.llm_chain = LLMChain(
            llm=llm,
            prompt=proposition_prompt

        propositions = [
            line.strip()
            for line in propositions_text.split("\n")
            if line.strip() and not line[0].isdigit()
        ]

명제 청킹의 장점.

  • 높은 검색 정확도. 각 명제는 명확한 의미를 가지므로, 특정 사실을 찾을 때 정확합니다.
  • 중복 제거. 같은 내용을 다르게 표현한 명제들을 자동으로 인식할 수 있습니다.
  • 추론 강화. 명제 간의 논리적 관계를 파악하기 쉬워 더 나은 추론이 가능합니다.

명제 청킹의 단점.

  • 높은 비용. LLM을 사용하므로 처리 시간과 비용이 많이 걭니다.
  • 품질 편차. LLM의 성능에 따라 추출된 명제의 질이 달라집니다.

기법 3: 의미 기반 청킹 (Semantic Chunking).

텍스트의 의미적 경계를 기반으로 청크를 나누는 기법입니다. 연속된 문장들의 의미적 유사성을 측정하여, 유사성이 급격히 떨어지는 지점에서 청크를 분할합니다.

Python
# 의미 기반 청킹 구현
import numpy as np
from langchain.embeddings import OpenAIEmbeddings

class SemanticChunker:
    def __init__(self, embeddings, threshold=0.5):
        self.embeddings = embeddings
        self.threshold = threshold

    def _cosine_similarity(self, vec1, vec2):
        # 코사인 유사도 계산
        return np.dot(vec1, vec2) / (np.linalg.norm(vec1) * np.linalg.norm(vec2))

    def chunk(self, text):
        # 문장으로 분할
        sentences = text.split(".")
        sentences = [s.strip() for s in sentences if s.strip()]

        # 각 문장의 임베딩 생성
        embeddings = [
            self.embeddings.embed_query(sentence)
            for sentence in sentences
        ]

        # 연속된 문장 간 유사도 계산
        chunks = [[sentences[0]]]
        for i in range(1, len(sentences)):
            similarity = self._cosine_similarity(embeddings[i-1], embeddings[i])

            # 유사도가 임계값보다 낮으면 새 청크 시작
            if similarity < self.threshold:
                chunks.append([sentences[i]])
            else:
                chunks[-1].append(sentences[i])

        # 청크 문자열로 변환
        return [".".join(chunk) + "." for chunk in chunks]

# 사용 예제
embeddings = OpenAIEmbeddings()
chunker = SemanticChunker(embeddings, threshold=0.5)

text = """대선 후보 A 의원이 오늘 기자회견을 열었습니다. 그는 새로운 경제 정책을 제시했습니다.
이 정책은 중소 기업 지원을 중심으로 합니다. 이와 별개로, 대선 후보 B는 교육 정책을 강조했습니다."""

semantic_chunks = chunker.chunk(text)
for i, chunk in enumerate(semantic_chunks):
    print(f"청크 {i+1}: {chunk}\n")
의미 기반 청킹의 특징. 의미 기반 청킹은 고정 크기 청킹과 달리, 문서의 의미 구조에 따라 동적으로 청크 크기가 결정됩니다. 복잡한 주제는 큰 청크로, 간단한 주제는 작은 청크로 분할됩니다.

기법 1: 쿼리 변환 (Query Transformations).

사용자의 질문을 다양한 형태로 변환하여, 여러 각도에서 관련 정보를 검색합니다.

변환 유형 1: 리라이팅.

# 쿼리 리라이팅
from langchain.llms import OpenAI
from langchain.chains import LLMChain
from langchain.prompts import PromptTemplate

# 쿼리 리라이팅 프롬프트
rewrite_prompt = PromptTemplate(
    input_variables=["query"],
    template="""다음 질문을 더 상세하고 구체적으로 리라이팅하세요.

원래 질문: {query}

더 상세한 질문:"""
)

class QueryTransformer:
    def __init__(self, llm):
        self.rewrite_chain = LLMChain(
            llm=llm,
            prompt=rewrite_prompt
        )

    def rewrite(self, query):
        # 원래 질문을 더 상세하게 변환
        rewritten = self.rewrite_chain.run(query=query).strip()
        return rewritten

    def decompose(self, query):
        # 복합 질문을 하위 질문으로 분해
        decompose_prompt = PromptTemplate(
            input_variables=["query"],
            template="""다음 복합 질문을 3-5개의 단순한 하위 질문으로 분해하세요.

원래 질문: {query}

하위 질문들:
1.
2.
3."""
        )
        decompose_chain = LLMChain(
            llm=self.llm,
            prompt=decompose_prompt
        )
        result = decompose_chain.run(query=query).strip()
        # 숫자와 점 제거
        questions = [
            line.strip() for line in result.split("\n")
            if line.strip() and not line[0].isdigit()
        ]
        return questions

# HyDE 구현
from langchain.llms import OpenAI
from langchain.prompts import PromptTemplate

class HyDERetriever:
    def __init__(self, llm, vectorstore, embeddings):
        self.llm = llm
        self.vectorstore = vectorstore
        self.embeddings = embeddings

    def retrieve(self, query, k=5):
        # 1단계: 가상 문서 생성
        hyde_prompt = PromptTemplate(
            input_variables=["query"],
            template="""다음 질문에 대한 완벽한 답변 문서를 작성하세요.

질문: {query}

답변 문서:"""
        )

        # 가상 문서 생성
        hypothetical_doc = self.llm.predict(
            query=query,
            **hyde_prompt.dict()
        )

        # 2단계: 가상 문서의 임베딩 계산
        doc_embedding = self.embeddings.embed_query(hypothetical_doc)

        # 3단계: 원래 질문의 임베딩 계산
        query_embedding = self.embeddings.embed_query(query)

        # 4단계: 두 임베딩을 모두 사용해 검색 (가중치 혼합)
        # 가상 문서를 기반으로 한 검색이 더 정확할 수 있음
        results = self.vectorstore.similarity_search_by_vector(
            doc_embedding,
            k=k
        )

        return {
            "hypothetical_doc": hypothetical_doc,
            "results": results
        }

# 사용 예제
hyde_retriever = HyDERetriever(llm, vectorstore, embeddings)

query = "최근 중국 경제 정책이 한국 수출에 미치는 영향"
results = hyde_retriever.retrieve(query)

print(f"가상 문서:\n{results['hypothetical_doc']}\n")
print(f"검색 결과 수: {len(results['results'])}")
HyDE의 강점. 질문이 불명확하거나 맥락이 부족할 때 특히 효과적입니다. 예를 들어, "정책의 영향"이라는 추상적 질문도 HyDE는 완전한 답변 문서를 생성해 관련 정보를 더 잘 찾아낼 수 있습니다.

# 복잡한 정치 질문 처리 시스템
class JournalismQueryEngine:
    def __init__(self, llm, vectorstore, embeddings):
        self.transformer = QueryTransformer(llm)
        self.hyde_retriever = HyDERetriever(llm, vectorstore, embeddings)
        self.llm = llm
        self.vectorstore = vectorstore

    def query(self, question):
        # 1단계: 쿼리 분해
        subquestions = self.transformer.decompose(question)

        # 2단계: 각 질문에 대해 HyDE로 검색
        all_results = []
        for subq in subquestions:
            results = self.hyde_retriever.retrieve(subq, k=3)
            all_results.extend(results["results"])

        # 3단계: 중복 제거
        unique_results = []
        seen = set()
        for r in all_results:
            if r.page_content not in seen:
                unique_results.append(r)
                seen.add(r.page_content)

        # 4단계: 최종 답변 생성
        context = "\n\n".join([r.page_content for r in unique_results[:5]])

        answer_prompt = PromptTemplate(
            input_variables=["context", "question"],
            template="""다음 정보를 기반으로 질문에 답하세요.

정보:
{context}

질문: {question}

답변:"""
        )

        answer = self.llm.predict(context=context, question=question)

        return {
            "original_question": question,
            "subquestions": subquestions,
            "retrieved_documents": unique_results,
            "answer": answer
        }

개요.

RAG 시스템의 성능은 검색된 텍스트의 품질에 크게 의존합니다. 컨텍스트 강화는 검색된 청크를 더 정보량 있고 명확하게 만들어 LLM이 더 정확한 답변을 생성하도록 돕습니다. 이 모듈에서는 청크의 컨텍스트를 개선하는 5가지 핵심 기법을 다룹니다.

각 청크에 문서 수준의 메타데이터와 계층적 정보를 헤더로 추가하는 기법입니다. 청크의 출처, 섹션, 날짜 등을 명시적으로 포함하면 LLM이 정보의 맥락을 더 정확히 이해합니다.

# 컨텍스트 헤더와 함께 청크 생성
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain.schema import Document

def add_contextual_headers(documents, metadata):
    """각 청크에 메타데이터 헤더 추가"""
    enhanced_docs = []

    for doc in documents:
        splitter = RecursiveCharacterTextSplitter(
            chunk_size=500,
            chunk_overlap=100
        )
        chunks = splitter.split_text(doc.page_content)

        for i, chunk in enumerate(chunks):
            header = f"[문서: {metadata['title']}] "
            header += f"[날짜: {metadata['date']}] "
            header += f"[섹션: {metadata['section']}]\n"

            enhanced_chunk = Document(
                page_content=header + chunk,
                metadata={**doc.metadata, 'chunk_id': i}
            )
            enhanced_docs.append(enhanced_chunk)

    return enhanced_docs

# 관련 세그먼트 동적 결합
def extract_relevant_segments(retrieval_results, context_size=2):
    """검색된 청크 주변의 관련 세그먼트 포함"""
    combined_segments = []

    for result in retrieval_results:
        chunk_id = result.metadata.get('chunk_id', 0)
        doc_id = result.metadata.get('doc_id')

        # 전후 청크 ID 계산
        before_ids = range(max(0, chunk_id - context_size), chunk_id)
        after_ids = range(chunk_id + 1, chunk_id + context_size + 1)

        # 세그먼트들을 순차적으로 결합
        segment_content = []
        for cid in list(before_ids) + [chunk_id] + list(after_ids):
            # 실제로는 벡터DB에서 해당 청크를 조회
            segment_content.append(result.page_content)

        combined = "\n".join(segment_content)
        combined_segments.append(combined)

    return combined_segments

# 문장 경계를 고려한 윈도우 확장
import re

def expand_context_window(text, chunk, window_chars=300):
    """청크 주변의 완전한 문장 포함"""
    chunk_start = text.find(chunk)

    # 이전 문장으로 확장
    before_start = max(0, chunk_start - window_chars)
    before_text = text[before_start:chunk_start]
    last_period = before_text.rfind('.')

    if last_period != -1:
        expanded_start = before_start + last_period + 1
    else:
        expanded_start = before_start

    # 이후 문장으로 확장
    chunk_end = chunk_start + len(chunk)
    after_end = min(len(text), chunk_end + window_chars)
    after_text = text[chunk_end:after_end]
    first_period = after_text.find('.')

    if first_period != -1:
        expanded_end = chunk_end + first_period + 1
    else:
        expanded_end = after_end

    expanded_chunk = text[expanded_start:expanded_end].strip()
    return expanded_chunk

# 판결문 분석 예시

# LLM으로 예상 질문 생성 및 문서 강화
def augment_documents_with_questions(documents, llm, num_questions=5):
    """각 문서에 대한 예상 질문 생성"""
    augmented_docs = []

    for doc in documents:
        prompt = f"""
다음 문서에 대해 저널리스트가 할 수 있는 {num_questions}개의 질문을 생성하세요.
질문들은 이 문서의 핵심 정보를 찾기 위한 것이어야 합니다.

문서:
{doc.page_content}

질문들:
"""

        response = llm.predict(prompt)
        questions = response.split('\n')[:num_questions]

        # 원본 문서에 질문들 추가
        augmented_content = doc.page_content + "\n\n[예상 질문들]\n"
        augmented_content += "\n".join(questions)

        augmented_doc = Document(
            page_content=augmented_content,
            metadata={**doc.metadata, 'augmented': True}
        )
        augmented_docs.append(augmented_doc)

    return augmented_docs

# 퓨전 검색: BM25 + 벡터 검색 결합
from langchain.retrievers import EnsembleRetriever
from langchain_community.retrievers import BM25Retriever
from langchain.vectorstores import FAISS

def create_fusion_retriever(documents, embeddings, k=3):
    """BM25와 벡터 검색을 결합한 퓨전 검색자"""

    # BM25 검색자 생성
    bm25_retriever = BM25Retriever.from_documents(
        documents,
        preprocess_func=lambda x: x.lower().split()
    )
    bm25_retriever.k = k

    # 벡터 검색자 생성
    vector_store = FAISS.from_documents(documents, embeddings)
    vector_retriever = vector_store.as_retriever(
        search_kwargs={'k': k}
    )

    # 앙상블 검색자: RRF로 순위 통합
    fusion_retriever = EnsembleRetriever(
        retrievers=[bm25_retriever, vector_retriever],
        weights=[0.5, 0.5]  # 가중치 조정 가능
    )

    return fusion_retriever

# 선거 보도 아카이브 검색
fusion_ret = create_fusion_retriever(documents, embeddings)
results = fusion_ret.get_relevant_documents(
    "2024년 대통령 선거 공약"
)

    filter_conditions = {
        "$and": [
            {"court": {"$eq": filters.get('court')}},
            {"case_type": {"$eq": filters.get('case_type')}},
            {"date": {"$gte": filters.get('start_date')}},
            {"date": {"$lte": filters.get('end_date')}}
        ]
    }

    # 필터 적용하여 검색
    results = retriever.similarity_search(
        query,
        k=10,
        filter=filter_conditions
    )

    return results

# 관련성과 다양성 최적화
def dartboard_retrieval(initial_results, k=5, diversity_weight=0.3):
    """관련성 + 다양성 기반 재순위화"""

    selected = []
    remaining = list(initial_results)

    # 관련성 점수가 가장 높은 것부터 시작
    selected.append(remaining.pop(0))

    while len(selected) < k and remaining:
        best_score = -1
        best_idx = 0

        for i, doc in enumerate(remaining):
            # 관련성 점수
            relevance = doc.metadata.get('score', 0)

            # 다양성 점수 (이미 선택된 것과의 거리)
            diversity = 1.0
            for selected_doc in selected:
                # 간단한 텍스트 유사도로 다양성 계산
                similarity = text_similarity(doc.page_content, selected_doc.page_content)
                diversity = min(diversity, 1.0 - similarity)

            # 결합 점수
            combined_score = relevance * (1 - diversity_weight) + diversity * diversity_weight

            if combined_score > best_score:
                best_score = combined_score
                best_idx = i

        selected.append(remaining.pop(best_idx))

    return selected

# 피드백 기반 검색 개선
import json
from datetime import datetime

class FeedbackAwareRetriever:
    """사용자 피드백으로 학습하는 검색자"""

    def __init__(self, base_retriever):
        self.base_retriever = base_retriever
        self.feedback_log = []  # 피드백 저장소
        self.query_weights = {}  # 쿼리별 가중치

    def retrieve(self, query):
        """기본 검색 수행"""
        results = self.base_retriever.get_relevant_documents(query)

        # 과거 피드백 기반 재순위화
        for_this_query = [fb for fb in self.feedback_log
                          if fb['query'] == query]

        if for_this_query:
            # 같은 쿼리의 과거 피드백으로 순위 조정
            results = self.rerank_by_feedback(results, for_this_query)

        return results

    def record_feedback(self, query, result_id, feedback, rating=5):
        """사용자 피드백 기록"""
        feedback_entry = {
            'query': query,
            'result_id': result_id,
            'feedback': feedback,  # "useful", "irrelevant", "partial"
            'rating': rating,  # 1-5 별점
            'timestamp': datetime.now().isoformat()
        }
        self.feedback_log.append(feedback_entry)

        # 피드백 기반 모델 가중치 조정
        self.update_weights(feedback_entry)

    def update_weights(self, feedback_entry):
        """피드백을 기반으로 모델 가중치 조정"""
        query = feedback_entry['query']

        if query not in self.query_weights:
            self.query_weights[query] = {
                'useful': 0,
                'irrelevant': 0,
                'total': 0
            }

        fb_type = feedback_entry['feedback']
        self.query_weights[query][fb_type] += 1
        self.query_weights[query]['total'] += 1

    def rerank_by_feedback(self, results, feedbacks):
        """과거 피드백 기반 재순위화"""
        scored_results = []

        for result in results:
            score = 1.0

            for fb in feedbacks:
                if fb['result_id'] == result.metadata.get('id'):
                    if fb['feedback'] == 'useful':
                        score *= 1.5  # 보너스
                    elif fb['feedback'] == 'irrelevant':
                        score *= 0.5  # 페널티

            scored_results.append((result, score))

        # 점수로 정렬
        sorted_results = sorted(scored_results, key=lambda x: x[1], reverse=True)

        return [r[0] for r in sorted_results]

# 다중 라운드 반복 검색
def iterative_research(initial_query, retriever, llm, rounds=3):
    """여러 라운드의 반복 검색으로 심층 조사"""

    research_log = {
        'initial_query': initial_query,
        'rounds': []
    }

    current_results = []
    current_query = initial_query

    for round_num in range(rounds):
        # 현재 쿼리로 검색
        results = retriever.get_relevant_documents(current_query)
        current_results.extend(results)

        round_info = {
            'round': round_num + 1,
            'query': current_query,
            'results_count': len(results),
            'results_summary': "\n".join([r.page_content[:200] for r in results])
        }
        research_log['rounds'].append(round_info)

        # 다음 라운드 쿼리 생성 (LLM이 자동으로)
        if round_num < rounds - 1:
            next_query_prompt = f"""
다음은 "{initial_query}"라는 주제에 대한 현재까지의 검색 결과입니다:

{round_info['results_summary']}

더 깊이 있는 조사를 위해, 위 결과를 바탕으로 다음에 찾아야 할 정보의 키워드를 제시하세요.
한 문장의 새로운 검색 쿼리를 생성하세요.
"""

            current_query = llm.predict(next_query_prompt)
            current_query = current_query.strip()

    return current_results, research_log

class JournalismFactChecker:
    """기자를 위한 종합 사실확인 봇"""

    def __init__(self, adaptive_retriever, feedback_retriever, llm):
        self.adaptive_ret = adaptive_retriever
        self.feedback_ret = feedback_retriever
        self.llm = llm
        self.fact_db = {}

    def verify_article_claims(self, article_text):
        """기사의 주요 주장들 검증"""

        # 1단계: LLM으로 주요 주장 추출
        extraction_prompt = f"""
다음 기사에서 검증이 필요한 구체적인 주장 5개를 추출하세요.
각각을 한 문장의 검색 쿼리로 변환하세요.

기사:
{article_text}

주장 목록:
"""

        claims = self.llm.predict(extraction_prompt).split('\n')

        # 2단계: 각 주장 검증
        verification_results = []

        for i, claim in enumerate(claims):
            if not claim.strip():
                continue

            # 적응형 검색으로 관련 자료 검색
            search_results, query_type = self.adaptive_ret.retrieve_adaptive(claim)

            # 3단계: LLM으로 검증 판단
            verify_prompt = f"""
다음 주장에 대해 검색된 자료들을 바탕으로 검증하세요.

주장: {claim}

검색 결과:
{chr(10).join([r.page_content[:300] for r in search_results[:3]])}

검증 결과:
- 정확함 / 부분 정확 / 부정확 / 확인 불가
- 관련 출처:
- 추가 설명:
"""

            verification = self.llm.predict(verify_prompt)

            result = {
                'claim_num': i + 1,
                'claim': claim,
                'verification': verification,
                'sources_count': len(search_results),
                'query_type': query_type
            }
            verification_results.append(result)

        return verification_results

기법 1. DeepEval.

DeepEval은 RAG 시스템을 종합적으로 평가하는 프레임워크입니다. 단순한 정확도를 넘어 정확성(Accuracy), 관련성(Relevance), 충실도(Faithfulness), 맥락 정확도(Context Precision) 등 6가지 메트릭을 한 번에 평가합니다.

기자 관점: "우리 AI 봇이 답한 내용이 정말 검색된 문서 내용과 일치하는가?" 이런 의문을 정량적으로 검증합니다.

DeepEval의 6가지 평가 지표.

  • 정확성(Accuracy): 답변이 실제 정답과 일치하는가.
  • 관련성(Relevance): 검색된 문서가 질문과 관련 있는가.
  • 충실도(Faithfulness): 답변이 검색된 문서 내용에 충실하는가.
  • 맥락 정확도(Context Precision): 상위 K개 검색 결과 중 관련 문서의 비율.
  • 맥락 회상률(Context Recall): 필요한 모든 정보를 검색했는가.
  • 응답 시간(Latency): 얼마나 빠른가.

기법 2. GroUSE.

GroUSE(Grounded Summarization Evaluation)는 LLM이 생성한 답변이 진짜 검색된 문서에 기반했는지 6가지 관점에서 평가합니다. 특히 "완전히 거짓", "부분적으로 참", "완전히 참"을 구분합니다.

GroUSE의 6가지 평가 항목.

  1. 사실성(Factuality): 답변의 각 문장이 출처에 있는가.
  2. 완전성(Completeness): 중요한 정보를 빠뜨리지 않았는가.
  3. 일관성(Consistency): 문장들 간 모순이 없는가.
  4. 관련성(Relevance): 불필요한 정보는 없는가.
  5. 유창성(Fluency): 문법적으로 자연스러운가.
  6. 구조(Structure): 답변이 논리적으로 정렬되었는가.

그래프 RAG.

학습 목표: 문서 간 관계를 그래프로 모델링하고, 엔티티·관계를 추출한 뒤 더 깊은 맥락을 활용한 RAG 시스템을 구축한다.

뉴스는 사람, 기관, 사건이 얽혀 있는 복잡한 네트워크입니다. 단순 키워드 검색으로는 "이 정치인과 저 의원이 왜 연결되어 있는가"같은 깊은 질문에 답할 수 없습니다. 그래프 RAG는 이러한 관계를 명시적으로 모델링하여 더 풍부한 맥락을 제공합니다.

기법 1. Graph RAG with Milvus.

Milvus는 벡터 검색과 그래프 검색을 함께 수행하는 벡터 데이터베이스입니다. 문서의 의미를 벡터로 변환하면서 동시에 문서 간 관계(엣지)를 저장합니다.

작동 원리.

문서 수집
↓
NLP로 엔티티·관계 추출
(서울시장, 예산안, 승인 등)
↓
벡터 임베딩 + 그래프 노드 생성
↓
Milvus에 저장 (벡터 + 그래프)
(검색 속도 최적화)
↓
하이브리드 쿼리
(벡터 검색 + 관계 탐색)
Python
# Milvus를 이용한 그래프 RAG
from pymilvus import connections, Collection
from spacy import load

# 한국어 NLP 모델 로드
nlp = load("ko_core_news_sm")

# Milvus 연결
connections.connect("default")

# 그래프 구축 함수
def build_graph_from_text(text):
    doc = nlp(text)
    entities = [ent.text for ent in doc.ents]

    # 간단한 관계 추출: (주어, 술어, 목어)
    relationships = []
    for token in doc:
        if token.dep_ == "nsubj":
            relationships.append((
                token.head.text, token.text, token.head.pos_
            ))

    return {
        "entities": entities,
        "relationships": relationships
    }

# 예: 뉴스 기사 분석
article = """
서울시장이 새로운 교통정책을 발표했습니다.
이 정책은 시의회에서 승인될 예정입니다.
"""

graph = build_graph_from_text(article)
print(f"엔티티: {graph['entities']}")
print(f"관계: {graph['relationships']}")

기법 3. Microsoft GraphRAG.

Microsoft GraphRAG는 문서로부터 자동으로 엔티티와 관계를 추출한 뒤 지식 그래프를 구축합니다. 추가로 커뮤니티 감지(Community Detection) 알고리즘을 적용해 관련 엔티티 그룹을 찾습니다.

예: 뉴스 말뭉치에서 "로비스트-의원 관계 네트워크"를 자동 생성합니다.

Microsoft GraphRAG의 3단계.

1단계: 엔티티·관계 추출.

LLM(예: GPT-4)이 문서를 읽고 "누가", "어떤 관계로", "누구와 연결되어 있는가"를 자동 추출합니다.

2단계: 지식 그래프 구축.

추출된 엔티티(노드)와 관계(엣지)로 그래프를 만듭니다.

3단계: 커뮤니티 감지.

Louvain 등 알고리즘으로 밀접하게 연결된 엔티티 그룹을 찾습니다. 예: "전직 로비스트" 커뮤니티, "보수진영 의원" 커뮤니티.

문제점 (일반 RAG).

키워드로 검색하면 "로비스트 A"와 "의원 B"의 관계가 여러 문서에 흩어져 있어 수동으로 연결해야 합니다.

해결책: GraphRAG.

기능 일반 RAG GraphRAG
로비스트 A와 B 찾기 각 문서 하나씩 한 번의 그래프 쿼리
A-B-C 경로 추적 불가능 그래프 경로 알고리즘
커뮤니티 발견 수동 자동 감지

결과.

커뮤니티 1: [로비스트A, 의원B, 의원C, 의원D]
       ↑ 모두 같은 건설사와 연결
커뮤니티 2: [로비스트X, 의원Y, 의원Z]
       ↑ 다른 진영의 로비 네트워크
기업 지배구조 파악: 대기업 임원진 간 이사회 관계, 자회사 계층도 등도 GraphRAG로 자동 추출 가능합니다.

핵심 정리.

  • Graph RAG with Milvus: 벡터 검색 + 그래프 검색 결합으로 의미와 관계를 동시에 활용.
  • Graph RAG with LangChain: Neo4j 등 그래프 DB와 연동, Cypher로 관계 조회.
  • Microsoft GraphRAG: LLM으로 자동 엔티티·관계 추출 → 지식 그래프 자동 구축 → 커뮤니티 감지.
  • 활용: 로비 네트워크, 기업 지배구조, 국회 관계도 등 복잡한 네트워크를 체계적으로 분석.

학습 목표: 검색 필요 여부를 판단하고, 결과를 자체 평가하고, 도구를 활용하는 에이전트 형태의 RAG를 구축한다.

이전 기법들은 "주어진 문서에서 답 찾기"에 집중했습니다. 고급 아키텍처는 한 걸음 더 나아갑니다. RAG 시스템이 스스로 판단합니다: "정말 문서를 검색해야 하나?", "이 답변이 맞나?", "웹을 또 뒤져야 하나?" 이런 지능형 의사결정이 가능한 아키텍처들을 소개합니다.

기법 1. RAPTOR.

RAPTOR(Recursive Abstractive Processing for Tree-Organized Retrieval)는 긴 문서를 계층적으로 정리합니다. 먼저 원본 문서를 작은 청크로 나누고, 각 청크를 요약합니다. 그 요약들을 또 요약해서... 최종적으로 "문서의 문서"인 트리 구조를 만듭니다.

RAPTOR의 계층 구조.

레벨 0 (원본)
└─ [문단1] [문단2] [문단3] [문단4] [문단5]

레벨 1 (청크 요약)
└─ [요약A] [요약B] [요약C]
   (문단1-2를 "주제A"로 요약)
   (문단3-4를 "주제B"로 요약)
   (문단5를 "주제C"로 요약)

레벨 2 (통합 요약)
└─ [통합 요약]
   ("주제A", "주제B", "주제C"를 하나의 핵심으로)

RAPTOR의 이점.

  • 빠른 검색: 여러 수준에서 검색 가능해 불필요한 세부사항 건너뜀.
  • 다양한 질문 대응: "전체 주제는?"(상위 레벨) vs "구체적인 숫자는?"(하위 레벨).
  • 맥락 유지: 요약 과정에서 중요 정보는 보존.

# RAPTOR 구현
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain.llms import OpenAI

# 1단계: 문서 청킹
splitter = RecursiveCharacterTextSplitter(
    chunk_size=1000,
    chunk_overlap=100
)

chunks = splitter.split_text(long_document)

# 2단계: 계층적 요약
llm = OpenAI(temperature=0)

def create_raptor_tree(chunks):
    current_level = chunks
    tree = [current_level]

    while len(current_level) > 1:
        next_level = []

        # 2개씩 묶어서 요약
        for i in range(0, len(current_level), 2):
            chunk_pair = current_level[i:i+2]
            summary_prompt = f"""
            다음 텍스트들을 한 문단으로 요약하세요:
            {' '.join(chunk_pair)}
            """
            summary = llm.predict(summary_prompt)
            next_level.append(summary)

        tree.append(next_level)
        current_level = next_level

    return tree

# 3단계: 검색
raptor_tree = create_raptor_tree(chunks)

def retrieve_raptor(query, raptor_tree):
    # 상위 레벨부터 검색
    for level in reversed(raptor_tree):
        best_match = find_most_similar(query, level)
        # 관련도 높으면 이 레벨의 자식 검색
    return best_match

기법 2. Self-RAG.

Self-RAG(Self-Reflective Retrieval-Augmented Generation)는 LLM이 스스로 판단합니다: "이 질문에 답하려고 문서를 검색해야 하나?" 또는 "나는 이미 알고 있어서 검색이 필요 없나?"

핵심 아이디어: RAG가 항상 검색하는 게 아니라, 필요할 때만 검색합니다. 이러면 속도도 빠르고 환각도 줄입니다.

Self-RAG의 4단계 판단 로직.

  1. 검색 필요성 판단(Retrieval Check): "이 질문을 답하려면 실제 문서가 필요한가?"
  2. 검색 실행 여부(If needed, retrieve): 필요하면 검색, 아니면 스킵.
  3. 답변 생성(Generate answer): 검색 결과 또는 기존 지식으로 답변.
  4. 자체 평가(Self-evaluate): "내 답변이 좋은가?" 아니면 더 검색할까?
Python
# Self-RAG 구현
from langchain.llms import OpenAI
from langchain.chains import LLMChain
from langchain.prompts import PromptTemplate

llm = OpenAI(temperature=0)

# 1단계: 검색 필요성 판단
retrieval_check_prompt = PromptTemplate(
    input_variables=["query"],
    template="""
    다음 질문에 답하려면 외부 문서가 필요한가?
    필요: YES, 불필요: NO
    질문: {query}
    """
)

class SelfRAG:
    def __init__(self, llm, retriever):
        self.llm = llm
        self.retriever = retriever

    def answer_query(self, query):
        # 1단계: 검색 필요성 판단
        need_retrieval = self.llm.predict(
            retrieval_check_prompt.format(query=query)
        )

        # 2단계: 필요하면 검색
        if "YES" in need_retrieval:
            context = self.retriever.get_relevant_documents(query)
        else:
            context = []

        # 3단계: 답변 생성
        answer_prompt = f"""
        질문: {query}
        맥락: {context}
        답변:
        """
        answer = self.llm.predict(answer_prompt)

        # 4단계: 자체 평가
        eval_prompt = f"""
        이 답변이 좋은가? (1-5)
        답변: {answer}
        """
        score = self.llm.predict(eval_prompt)

        return {
            "answer": answer,
            "retrieval_needed": need_retrieval,
            "self_score": score
        }

rag = SelfRAG(llm, retriever)
result = rag.answer_query("한국 현대 미술의 주요 특징은?")

기법 3. Corrective RAG (CRAG).

Corrective RAG는 검색 결과를 자체 평가하고, 관련도가 낮으면 웹 검색으로 보강합니다. "벡터 DB의 문서로 부족하면 구글 검색"이라는 논리입니다.

주의: 웹 검색은 비용과 속도가 문제입니다. 필요한 경우에만 활용해야 합니다.

CRAG의 피드백 루프.

단계 작업 결정
1. 검색 벡터 DB에서 문서 검색 관련도 평가
2. 평가 검색 결과가 관련 있는가? YES: 진행 / NO: 웹 검색
3. 보강 필요시 웹 검색 결과 합치기
4. 생성 최종 맥락으로 답변 사용자에게 제시
Python
# Corrective RAG 구현
from langchain.retrievers.web_research import WebResearchRetriever
from langchain.utilities import GoogleSearchAPIWrapper

class CorrectiveRAG:
    def __init__(self, llm, vector_retriever, web_retriever):
        self.llm = llm
        self.vector_retriever = vector_retriever
        self.web_retriever = web_retriever

    def retrieve_and_correct(self, query):
        # 1단계: 벡터 DB 검색
        vector_docs = self.vector_retriever.get_relevant_documents(query)

        # 2단계: 관련도 평가
        relevance_scores = [doc.metadata.get("score", 0) for doc in vector_docs]
        avg_score = sum(relevance_scores) / len(relevance_scores) if relevance_scores else 0

        if avg_score < 0.6:  # 임계값
            # 3단계: 웹 검색으로 보강
            web_docs = self.web_retriever.get_relevant_documents(query)
            all_docs = vector_docs + web_docs
        else:
            all_docs = vector_docs

        return all_docs

rag = CorrectiveRAG(llm, vector_retriever, web_retriever)
docs = rag.retrieve_and_correct("2024 국회의원 선거 최신 뉴스")

기법 4. Agentic RAG.

Agentic RAG는 RAG를 단순한 도구(tool)로 보고, 에이전트(agent)가 RAG를 포함한 여러 도구를 능동적으로 선택합니다. 검색, 계산, 웹 조회, 파일 읽기 등을 조합합니다.

Agentic RAG의 도구 조합.

  • RAG 도구: 내부 문서 검색.
  • 웹 검색 도구: 실시간 뉴스 또는 최신 정보.
  • 계산 도구: 수치 연산.
  • 파일 도구: CSV, PDF 읽기.
  • API 도구: 외부 서비스 연동.

    )
]

# 에이전트 초기화
agent = initialize_agent(
    tools,
    llm,
    agent="zero-shot-react-description",
    verbose=True
)

# 사용 예: 복합 질문
result = agent.run(
    """
    2024년 한국 경제 성장률을 찾고,
    이를 지난해와 비교하세요.

    """
)
프로덕션급 예시: 탐사보도 에이전트가 "정치인 A의 자산변동 추적"을 받으면, 먼저 내부 기사(RAG)로 이전 기사를 찾고, 부족하면 뉴스 DB를 검색하고, 필요하면 공시 자료를 PDF에서 읽고, 최종적으로 통합 보도자료를 생성합니다.

실제 사용 사례: 실시간 뉴스 팩트체크 시스템.

시나리오: "정치인이 거짓 주장".

어떤 정치인이 "올해 실업률이 3%로 사상 최저"라고 주장합니다. 기자는 이를 즉시 검증해야 합니다.

이전 방식 (수동).

  1. 통계청 사이트 방문
  2. 최신 실업률 수치 찾기
  3. 역사 데이터와 비교
  4. 기사 작성

RAPTOR + Self-RAG + Agentic의 통합 사용.

입력: "정치인 주장: 올해 실업률 3% 사상 최저"

단계 1 (Self-RAG): "이 주장은 팩트체크가 필요한가?" → YES

단계 2 (Agentic): 에이전트가 도구 선택

→ RAG 도구: 기존 실업률 기사 검색

→ Web 도구: 통계청 최신 자료 수집

→ Calc 도구: "3% vs 역대 최저치" 비교

단계 3 (RAPTOR): 장문 통계 자료를 계층적으로 요약

출력: "주장은 거짓입니다. 올해 실업률은 3.2%이며, 2008년 2.8%가 최저입니다."

기자 활용.

5분 내에 자동 팩트체크 결과를 받고, 추가 취재만 진행합니다.

핵심 정리.

  • RAPTOR: 긴 문서를 계층적 트리로 정리해 빠른 검색 + 맥락 유지.
  • Self-RAG: "정말 검색이 필요한가?"를 LLM이 판단 → 속도 + 환각 감소.
  • Corrective RAG: 벡터 DB 부족 시 자동으로 웹 검색 보강.
  • Agentic RAG: RAG + 웹 + 계산 + API 등을 에이전트가 조합해 복합 질문 해결.
  • 뉴스룸 활용: 팩트체크, 탐사보도 리서치, 실시간 뉴스 검증을 자동화.

퀴즈.

Self-RAG와 Corrective RAG의 핵심 차이는?

Self-RAG는 "검색 자체가 필요한가?"를 판단하고, Corrective RAG는 "현재 검색 결과가 충분한가?"를 평가하고 부족하면 웹 검색으로 보강합니다.
