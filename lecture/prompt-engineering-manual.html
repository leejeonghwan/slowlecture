Messages API 기본 구조.

Claude API의 핵심은 messages.create() 메서드입니다. 이 메서드에는 다음 주요 파라미터가 있습니다:

파라미터 설명 예시
model 사용할 Claude 모델 버전. claude-sonnet-4-20250514
max_tokens 응답의 최대 길이 (토큰 단위). 1024
messages 대화 히스토리 (role과 content). [{"role": "user", "content": "..."}]
system Claude의 역할을 정의하는 시스템 프롬프트 (선택). "당신은 기자입니다."

첫 번째 API 호출.

간단한 질문을 Claude에게 던져봅시다.

Python 
message = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=1024,
    messages=[
        {"role": "user", "content": "대한민국의 수도는?"}
    ]
)

print(message.content[0].text)

Output

모델 선택과 파라미터 조정.

Claude는 여러 버전이 있으며, 각각 속도와 능력이 다릅니다:

모델 특징 용도
claude-sonnet-4-20250514 빠르고 효율적, 토큰당 비용 낮음. 빠른 응답이 필요한 작업, 대량 처리.
claude-opus-4-20250514 가장 강력하고 정확한 모델. 복잡한 분석, 정밀도가 중요한 작업.

temperature 파라미터로 응답의 창의성을 조절할 수 있습니다 (0.0 ~ 1.0):

  • 0.0: 완전히 결정적, 같은 입력에 항상 같은 출력.
  • 0.5: 균형잡힌 설정 (기본값).
  • 1.0: 최대 창의성, 매번 다른 응답.
Python 
# 정밀도가 높은 설정
message = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=1024,
    temperature=0.2,
    messages=[
        {"role": "user", "content": "대한민국의 수도는?"}
    ]
)

헬퍼 함수 만들기.

반복적으로 API를 호출하는 것을 간편하게 하기 위해 헬퍼 함수를 만들어봅시다. 이렇게 하면 코드가 깔끔하고 재사용 가능해집니다.

Python 
def get_completion(prompt, system="", model="claude-sonnet-4-20250514", max_tokens=2048):
    """
    Claude API를 호출하는 헬퍼 함수.

    Args:
        prompt: 사용자 질문 또는 요청.
        system: 시스템 프롬프트 (선택사항).
        model: 사용할 모델 (기본값: Sonnet).
        max_tokens: 최대 응답 길이.

    Returns:
        Claude의 응답 텍스트.
    """
    kwargs = {
        "model": model,
        "max_tokens": max_tokens,
        "messages": [{"role": "user", "content": prompt}]
    }

    if system:
        kwargs["system"] = system

    message = client.messages.create(**kwargs)
    return message.content[0].text


# 사용 예
result = get_completion("서울의 인구는?")
print(result)

Chapter 2: 명확하고 직접적인 지시.

35분 초급

프롬프트의 품질을 크게 좌우하는 것은 "명확함"입니다. 같은 내용도 어떻게 전달하느냐에 따라 결과가 완전히 달라집니다.

프롬프트는 구체적일수록 좋다.

모호한 지시는 예상 밖의 결과를 낳습니다. 구체적이고 세부적인 요청을 하면 원하는 결과를 얻을 가능성이 높아집니다.

나쁜 예.

"기사 써줘."

문제: 주제, 길이, 형식, 대상 독자 모두 불명확함.

좋은 예.

"서울시 2025년 예산안에 대한 300자 스트레이트 기사를 써줘. 핵심 수치와 전년 대비 증감률을 포함해줘."

개선점: 주제, 길이, 형식, 포함 요소가 명확함.

모호한 지시 vs 명확한 지시 비교.

다양한 상황에서 지시의 명확성이 중요한 영향을 미칩니다:

모호한 요청.

"이 보도자료를 기사로 만들어줘."

명확한 요청.

"이 보도자료를 뉴스통신사 스타일의 500자 스트레이트 기사로 변환해줘. 역피라미드 구조를 사용하고, 핵심 정보를 첫 문장에 포함시켜줘."

모호한 요청.

"기사 분석해줘."

명확한 요청.

"다음 기사를 분석해줘:\n1. 기자의 입장(찬성/반대/중립)\n2. 사용된 논거와 출처\n3. 빠진 관점\n각 항목은 3-4문장으로."

단계별 지시 (Numbered Steps).

복잡한 작업은 단계를 나누어 지시하면 더 명확합니다.

Python 
prompt = """

2단계: 사실들을 중요도순으로 정렬해.
3단계: 역피라미드 구조로 200자 기사를 다시 작성해.

[기사 본문]
서울시가 2025년 예산안에서 저출산 대응에 역대 최대 규모인
3조 5천억 원을 배정했다. 이는 전년 대비 25% 증가한 규모로,
출산 장려금 인상과 육아 시설 확충에 중점을 두고 있다...
"""

result = get_completion(prompt)
print(result)

주의사항.

Chapter 5. 출력 형식 제어.

40분 중급
구조화 된 출력 XML JSON 파싱

왜 형식이 중요한가?.

핵심 아이디어. 올바른 출력 형식은 Claude의 응답을 자동으로 처리하고 다른 시스템에 연동할 수 있게 합니다. 뉴스룸에서는 기사 메타데이터를 자동으로 추출하고, CMS에 바로 입력할 수 있습니다.

Claude의 응답이 자유로운 텍스트가 아니라 구조화된 형식이면 다음이 가능합니다.

  • 후속 처리(파싱) 가능한 출력: Python, JavaScript 등으로 데이터를 자동 추출.
  • 뉴스룸 시스템과의 연동: 기사 제목, 요약, 카테고리를 CMS에 직접 입력.
  • 품질 보증: 형식이 정해지면 오류 감지가 쉬워집니다.

Chapter 6. 단계적 사고 (Chain of Thought).

40분 중급
추론 Thinking 태그 팩트체크 복잡한 분석

Claude에게 "생각할 시간"을 주기.

핵심 아이디어. 바로 답을 요구하면 Claude가 실수할 확률이 높습니다. "먼저 생각하고 답하세요"라는 지시를 주면 정확도가 크게 올라갑니다.

복잡한 문제에서 Claude는 단계적 추론이 필요합니다.

  • 기사 신뢰성 평가: 출처 확인, 수치 검증, 논리적 일관성을 단계적으로 분석.
  • 팩트체크: 여러 각도에서 검증해야 정확합니다.
  • 데이터 해석: 통계 수치의 의미를 단계별로 분석.

Zero-Shot vs Few-Shot.

핵심 아이디어. 예시를 보여주면 Claude가 패턴을 학습하고 더 정확하게 응답합니다. 뉴스룸에서 기사 톤이나 스타일을 일관되게 유지할 때 유용합니다.

두 가지 접근 방식을 비교합니다.

  • Zero-Shot: 예시 없이 바로 요청. 빠르지만 부정확할 수 있습니다.

산업별 프롬프트 설계는 단순히 명령을 입력하는 것이 아니라 체계적인 프로세스입니다. 특정 업계의 요구사항을 정확히 파악하고, 이를 Claude가 이해할 수 있는 구조로 변환한 후, 실제 운영 데이터로 반복 검증하는 과정을 거칩니다.

이 단계들은 다음과 같이 진행됩니다:

  • 요구사항 분석: 실제 업무에서 필요한 입력, 출력, 평가 기준을 명확히 정의합니다.
  • 구조 설계: 시스템 프롬프트와 유저 프롬프트, 출력 형식을 설계합니다.
  • 반복 개선: 테스트 결과를 바탕으로 프롬프트를 수정하고 성능을 추적합니다.

프롬프트 체이닝.

40분 고급 파이프라인

체이닝이란?

정의: 복잡한 작업을 여러 개의 작은 프롬프트로 나누고, 각 프롬프트의 출력을 다음 프롬프트의 입력으로 전달하는 방식입니다.

체이닝은 다음과 같은 구조를 갖습니다:

  • 1단계: 입력 데이터 처리 → 출력 A
  • 2단계: 출력 A를 입력으로 받음 → 출력 B
  • 3단계: 출력 B를 입력으로 받음 → 최종 결과

왜 체이닝이 필요한가?

하나의 거대한 프롬프트로 모든 작업을 처리하려면 여러 문제가 발생합니다:

단일 프롬프트의 한계:
  • 프롬프트가 너무 길면 Claude의 성능이 하락합니다.
  • 어느 단계에서 오류가 발생했는지 파악하기 어렵습니다.
  • 각 단계를 독립적으로 개선할 수 없습니다.
  • 특정 단계에서 검증하기 어렵습니다.

반면 체이닝을 사용하면:

  • 각 단계가 명확하고 이해하기 쉽습니다.
  • 각 단계마다 결과를 검증할 수 있습니다.
  • 오류가 발생한 단계를 정확히 파악할 수 있습니다.
  • 각 단계를 독립적으로 개선할 수 있습니다.

    similarities = cosine_similarity([query_embedding], embeddings)[0]

    # 상위 3개 문서 반환
    top_indices = similarities.argsort()[-3:][::-1]
    return [documents[i] for i in top_indices]

청킹(Chunking) 전략.

문서를 적절한 크기로 나누는 것이 RAG의 핵심입니다.

Python 
# 의미 단위로 청킹
def chunk_by_paragraph(text):
    """문단 단위로 청킹합니다."""
    paragraphs = text.split("\n\n")
    chunks = []
    
    for para in paragraphs:
        # 문단이 너무 길면 문장 단위로 추가 분할
        if len(para) > 500:
            sentences = para.split(".")
            chunk = ""
            for sent in sentences:
                chunk += sent + "."
                if len(chunk) > 300:
                    chunks.append(chunk)
                    chunk = ""
            if chunk:
                chunks.append(chunk)
        else:
            chunks.append(para)
    
    return chunks

chromadb로 간단한 RAG 구현.

chromadb는 로컬 벡터 데이터베이스 구축에 가장 간단한 도구입니다.

Python 
import chromadb

# 클라이언트 및 컬렉션 생성
client = chromadb.Client()
collection = client.create_collection("articles")

# 기사를 벡터 DB에 저장
collection.add(
    documents=["기사1 본문...", "기사2 본문...", "기사3 본문..."],
    ids=["article_1", "article_2", "article_3"],
    metadatas=[
        {"date": "2024-03-20", "category": "정치"},
        {"date": "2024-03-19", "category": "경제"},
        {"date": "2024-03-18", "category": "사회"}
    ]
)

# 관련 기사 검색
results = collection.query(
    query_texts=["부동산 정책 변화"],
    n_results=3
)

# 검색 결과 활용
for doc in results["documents"][0]:
    print(doc)

RAG의 한계와 주의점.

RAG 사용 시 주의점:
  • 검색 품질이 곧 답변 품질: 좋은 문서를 찾지 못하면 좋은 답변도 나올 수 없습니다.
  • 오래된 정보 문제: 정기적으로 DB를 업데이트해야 합니다.
  • 토큰 한도 관리: 너무 많은 검색 결과를 전달하면 토큰을 낭비합니다.
  • 환각 가능성: 검색 결과가 없어도 Claude가 답변할 수 있으므로, "자료에 없음"이라고 명시적으로 지시하세요.

정답: B번

RAG에서는 검색 품질이 답변 품질을 결정합니다. 아무리 좋은 모델이라도 관련 없는 문서만 검색되면 도움이 되지 않습니다. 따라서 좋은 청킹 전략, 임베딩 모델, 검색 알고리즘이 핵심입니다.
